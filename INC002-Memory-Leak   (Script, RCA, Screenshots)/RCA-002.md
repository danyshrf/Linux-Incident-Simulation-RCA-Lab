# 📝 Root Cause Analysis: INC-002 (Memory Leak / Runaway Process)

**Lab Project:** KVM Monitoring Stack  
**Affected System:** VM3 — Target Server (`192.168.200.10`)  
**Severity:** HIGH  

---

## 1. Incident Summary
On the simulated production environment, the RAM utilization on VM3 exceeded the 85% usage threshold, triggering a critical monitoring alert. 

* **Incident ID:** INC-002
* **Detection Method:** Grafana Alert: "High RAM Alert" (>85% threshold).
* **Reported By:** Grafana Alerting → Email Notification.

---

## 2. Impact
* **System Health:** System responsiveness degraded as available memory approached zero. There was a high risk of the OOM (Out-of-Memory) Killer activating and terminating critical processes.
* **Monitoring:** Node Exporter and essential system services remained functional during the incident, allowing for continuous telemetry.

---

## 4. Technical Root Cause
The incident was deliberately caused by a memory stress tool (`stress-ng`) allocating 80% of available RAM using 2 concurrent workers.

In a production environment, this scenario maps to common real-world failures such as:
* Application memory leaks where object references are not released and the heap grows unbounded.
* Java applications running with insufficient or missing heap limits (`-Xmx`).
* Database buffer pools consuming all available RAM.
* Multiple processes simultaneously holding large in-memory caches.
* Containers operating without defined memory limits on a shared host.

---

## 5. Investigation & Remediation

### Diagnostic Path
I utilized the following commands to isolate the memory exhaustion:
1. **Memory Overview:** Checked `free -h` for a human-readable summary and `cat /proc/meminfo` for a detailed kernel breakdown.
2. **Finding the Culprit:** Used `ps aux --sort=-%mem | head -15` and interactive `top` (sorted by memory) to find the offending process.
3. **Deep Inspection:** Analyzed `cat /proc/<PID>/status | grep -E "VmRSS|VmPeak|VmSize"` and checked `ls -la /proc/<PID>/exe` to identify the binary.
4. **OOM Killer Audit:** Verified the kernel state using `dmesg | grep -i "oom"` and `journalctl -k | grep -i "oom"`.

### Recovery Action
| Step | Action | Command |
| :--- | :--- | :--- |
| 1 | Identify top memory process | `ps aux --sort=-%mem | head -5` |
| 2 | Immediate Remediation | `pkill stress-ng` |
| 3 | Verify memory released | `free -h` |
| 4 | Audit OOM killer activity | `dmesg | grep oom` |
| 5 | Confirm alert resolved | Checked Grafana → Alerting → Alert Rules |

---

## 6. Monitoring & Alert Behavior
The incident triggered based on the following PromQL query:
`(1 - (node_memory_MemAvailable_bytes / node_memory_MemTotal_bytes)) * 100 > 85`

* The "Memory Usage %" panel showed a steep upward climb in the line graph.
* The alert state progressed linearly: Normal → Pending → Firing.
* After executing the `pkill` command, memory returned to baseline within 10 seconds, and the alert auto-resolved.

---

## 7. Preventive Measures
To prevent runaway memory consumption in production, I recommend the following infrastructure changes:
* **Resource Quotas:** Set strict memory limits on all containers and application processes, utilizing `cgroups` to enforce hard limits per service.
* **Tiered Alerting:** Establish an early warning alert tier at 75% RAM before hitting the critical 85% threshold.
* **App-Level Telemetry:** Implement application-level memory monitoring with automated heap dump triggers.
* **Buffer Allocation:** Configure appropriate swap space as an emergency buffer (while noting this is not a substitute for resolving underlying memory leaks).
