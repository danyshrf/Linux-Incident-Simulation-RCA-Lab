# Root Cause Analysis: INC-001 (Disk Saturation)

- **Lab Project:** Linux Incident Simulation & RCA Lab  
- **Affected System:** VM3 — Target Server (192.168.200.10) 
- **Severity:** HIGH 


---

## 1. Incident Summary
On the simulated production environment, the root filesystem on VM3 exceeded the 85% usage threshold, triggering a critical monitoring alert. 

* **Incident ID:** INC-001 [cite: 5]
* **Detection Method:** Grafana Alert: "Disk Almost Full Alert" (>85% threshold). 
* **Duration:** Approximately 5 minutes from detection to resolution. 
* **Notification:** Alert routed via Grafana Alerting to Email. 

---

## 2. Impact
* **System Health:** Disk usage exceeded 85%, leaving minimal overhead for system operations. 
* **Monitoring:** Node Exporter remained active, providing a continuous data stream for the RCA. 
* **Risk:** Potential for service failure or system hang if the root filesystem reached 100% capacity.

## 4. Technical Root Cause
The incident was triggered by a runaway file-writing process that generated multiple 500MB dummy files in the `/tmp/incident-test/` directory.

In a production environment, this mimics:
* **Log Rotation Failure:** Logs accumulating without being purged. 
* **Unconstrained Backups:** Local backup jobs writing to the root partition. 
* **Application Errors:** Temporary files from crashed processes failing to clear. 

---

## 5. Investigation & Remediation

### Diagnostic Path
I utilized the following commands to isolate the issue:
1. **Initial Check:** `df -h` to identify the saturated mount point. 
2. **Directory Analysis:** `du -h / --max-depth=2 | sort -rh` to locate the largest directories. 
3. **Deep Dive:** `ls -lh /tmp/incident-test/` to confirm the presence of large, unauthorized files. 

### Recovery Action
| Step | Action | Command |
| :--- | :--- | :--- |
| 1 | Locate culprit files |`du -sh /tmp/*` |
| 2 | Immediate Remediation | `rm -rf /tmp/incident-test/`  |
| 3 | Verify Recovery | `df -h /`  |

---

## 6. Preventive Measures
To prevent a recurrence of this incident, the following steps will be implemented:
* **Log Management:** Configure `logrotate` for all application and system log paths. 
* **Tiered Monitoring:** Set a **Warning** alert at 75% and a **Critical** alert at 85% to allow for proactive intervention.
* **Automated Cleanup:** Implement `systemd-tmpfiles` to manage `/tmp` data automatically. 
* **Automation:** Deploy a cron job to automatically audit and clean old backup/dump files.
